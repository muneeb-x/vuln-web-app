## 0. Plan Overview

This plan implements the bcrypt fix specified in [bcrypt-password-hashing.md](./bcrypt-password-hashing.md). It closes **Vulnerability #5 (Weak Password Storage)** and **only** that vulnerability. The work is split into **six phases** so each step is small, individually verifiable, and easy to revert.

The seven remaining intentional vulnerabilities (VULN-1 SQLi, VULN-2 Stored XSS, VULN-3 Reflected XSS, VULN-4 Session Hijacking, VULN-6 Exposed DB, VULN-7 No Rate Limiting, VULN-8 No CSRF) MUST remain exploitable after every phase. Each phase ends with an explicit "MUST NOT" callout listing things that would silently close another vulnerability.

## Phase 2 — Rewrite `backend/app/core/security.py` to Use Bcrypt

### 2.1 Goal

Replace the entire body of `security.py` with the bcrypt-based implementation. Preserve the public function names and signatures so existing call-sites (`auth_service.signup()` already imports `hash_password`; `auth_service.login()` will import `verify_password` in Phase 3) keep working.

### 2.2 File to Modify

- `backend/app/core/security.py`

### 2.3 New Content (Whole-File Replacement)

The file is currently 9 lines. Replace its full content with:

```python
import bcrypt

BCRYPT_ROUNDS = 12


def hash_password(password: str) -> str:
    salt = bcrypt.gensalt(rounds=BCRYPT_ROUNDS)
    return bcrypt.hashpw(password.encode("utf-8"), salt).decode("utf-8")


def verify_password(plain: str, hashed: str) -> bool:
    try:
        return bcrypt.checkpw(plain.encode("utf-8"), hashed.encode("utf-8"))
    except Exception:
        return False
```

### 2.4 Line-by-Line Justification

| Line | Decision | Spec ref |
|------|----------|----------|
| `import bcrypt` | New runtime dependency declared in Phase 1 | FR-06 |
| `BCRYPT_ROUNDS = 12` | Named module-level constant — auditable in code review; monkey-patchable in tests | FR-03, NFR-05 |
| `password.encode("utf-8")` | Consistent UTF-8 encoding; supports non-ASCII passwords (EC-03) | NFR-06 |
| `bcrypt.gensalt(rounds=BCRYPT_ROUNDS)` | Fresh per-call salt — two identical plaintexts yield distinct hashes | FR-01, SP-05, AC-03 |
| `bcrypt.hashpw(...).decode("utf-8")` | Return `str` (matches `-> str` signature) so the SQL `INSERT` still gets a string | NFR-02, FR-05 |
| `try: bcrypt.checkpw(...)` | Bcrypt raises `ValueError` on malformed `hashed` (e.g. legacy 32-char MD5 hex) — catching prevents process crash | FR-02, EC-01, EC-05, AC-05 |
| `except Exception: return False` | Collapses all failure modes (`ValueError`, `TypeError` on `None`, etc.) to `False`. No logging — no information leakage. | NFR-04 |

### 2.5 What NOT to Change in Phase 2

- **DO NOT** keep the old `import hashlib` line. It becomes dead and should be removed cleanly (a stray import would not break behavior, but it would be a misleading footprint).
- **DO NOT** add an MD5 fallback path inside `verify_password()`. The spec is explicit (FR-02, EC-01): legacy MD5 returns `False`, not a re-hash, not a parallel-MD5 check. Adding fallback authentication for legacy rows would silently re-open VULN-5.
- **DO NOT** add a "rehash on successful login" upgrade path. The spec (§11) deliberately rejects it: you cannot accept a password against an MD5 hash, validate it, and then store a bcrypt re-hash, because step one requires authenticating with code we are removing.
- **DO NOT** add `logger.warning("legacy hash detected")` or any other logging. NFR-04 forbids information leakage; for the educational lab, no logging at all is cleaner than a redacted one.
- **DO NOT** introduce a `peppered` variant, key-stretching pre-hash, HKDF, or anything other than vanilla bcrypt. NFR-07 explicitly accepts bcrypt's 72-byte truncation behavior.
- **DO NOT** change the function names or signatures. `hash_password(password: str) -> str` and `verify_password(plain: str, hashed: str) -> bool` are pinned by NFR-02 / AC-01.

## Phase 3 — Rewrite `auth_service.login()` for Python-Side Password Comparison

### 3.1 Goal

Because bcrypt produces a different output for the same password each call, the legacy login pattern `WHERE username = '<u>' AND password = '<hash>'` cannot match. Replace the password-in-SQL strategy with **fetch-by-username, then `verify_password()` in Python**. Critically: **do not** parameterize the username branch — the `WHERE username = '" + username + "'` concatenation MUST remain to preserve VULN-1 (spec §2.3, AC-07).

### 3.2 File to Modify

- `backend/app/services/auth_service.py`

### 3.3 Import Update

**Before** (L7):

```python
from app.core.security import hash_password
```

**After**:

```python
from app.core.security import hash_password, verify_password
```

`hash_password` is still imported because `signup()` continues to use it. `verify_password` is added for the new login flow.

### 3.4 `signup()` — Body Unchanged

The `signup()` function (L10–38) is **NOT touched** in this phase. Its `hashed = hash_password(password)` line now produces a bcrypt string instead of an MD5 hex — that is the entire visible change to its behavior, but the source is not edited. Specifically:

- The string-concatenated `INSERT` query stays exactly as it is (VULN-1 preservation).
- The `sqlite3.IntegrityError` handling stays.
- The generic `Exception` handler stays.
- The `finally: conn.close()` stays.

### 3.5 `login()` — Targeted Rewrite

The current `login()` body (L41–74) is replaced as follows.

**Before** (L41–74):

```python
def login(request: Request, username: str, password: str):
    if not username or not password:
        return JSONResponse(
            content={"error": "Username and password are required"},
            status_code=401,
        )

    hashed = hash_password(password)

    # VULNERABILITY #1: SQL Injection via string concatenation
    query = "SELECT * FROM users WHERE username = '" + username + "' AND password = '" + hashed + "'"

    conn = get_db()
    try:
        cursor = conn.execute(query)
        user = cursor.fetchone()
    except Exception:
        return JSONResponse(
            content={"error": "Invalid username or password"},
            status_code=401,
        )
    finally:
        conn.close()

    if user:
        request.session["user_id"] = user["id"]
        request.session["username"] = user["username"]
        request.session["email"] = user["email"]
        return JSONResponse(content={"success": True, "redirect": "/welcome"})
    else:
        return JSONResponse(
            content={"error": "Invalid username or password"},
            status_code=401,
        )
```

**After**:

```python
def login(request: Request, username: str, password: str):
    if not username or not password:
        return JSONResponse(
            content={"error": "Username and password are required"},
            status_code=401,
        )

    # VULNERABILITY #1: SQL Injection via string concatenation
    # (Password comparison is performed in Python — see verify_password below —
    # because bcrypt hashes cannot be matched with an SQL equality check.
    # The username branch is intentionally still concatenated to preserve VULN-1.)
    query = "SELECT * FROM users WHERE username = '" + username + "'"

    conn = get_db()
    try:
        cursor = conn.execute(query)
        user = cursor.fetchone()
    except Exception:
        return JSONResponse(
            content={"error": "Invalid username or password"},
            status_code=401,
        )
    finally:
        conn.close()

    if user and verify_password(password, user["password"]):
        request.session["user_id"] = user["id"]
        request.session["username"] = user["username"]
        request.session["email"] = user["email"]
        return JSONResponse(content={"success": True, "redirect": "/welcome"})
    else:
        return JSONResponse(
            content={"error": "Invalid username or password"},
            status_code=401,
        )
```

### 3.6 Diff Summary

Three logical changes inside `login()`:

1. **Remove** the `hashed = hash_password(password)` line (no longer needed; password is verified in Python against the stored bcrypt hash).
2. **Rewrite** the `query` string to drop the `AND password = '" + hashed + "'` suffix. Keep the `WHERE username = '" + username + "'` concatenation byte-for-byte (VULN-1).
3. **Replace** the truthy `if user:` branch with `if user and verify_password(password, user["password"]):`. The session writes and the JSON success payload are unchanged.

The error path (no row, or SQL error, or password mismatch) returns the **same** JSON 401 with the **same** error text — preserving FR-04's "no information leakage between cases" requirement.

### 3.7 What NOT to Change in Phase 3

- **DO NOT** parameterize the `SELECT` query. `conn.execute(query, (username,))` would close VULN-1 and is out of scope (spec §2.3, AC-07).
- **DO NOT** add a `LIMIT 1` clause to "harden" the query against multi-row injection — that would change the SQL injection behavior surface.
- **DO NOT** add a `bcrypt.checkpw` import directly here. The comparison MUST go through `verify_password()` so the `try/except` legacy-MD5 protection is centralised.
- **DO NOT** rewrite `signup()`. Its body stays byte-identical (FR-05).
- **DO NOT** change response shape, status codes, or error text. The spec's success/failure JSON contract is identical to before.
- **DO NOT** add timing-attack mitigation (constant-time response delay). Bcrypt's own ~250 ms cost dominates timing noise for the password-match branch; for the no-user branch the spec does not require constant time (FR-04 last sentence).
- **DO NOT** remove the `# VULNERABILITY #1: SQL Injection ...` comment. It is documentation for students. Update the comment text to reflect that the password branch is gone (see "After" snippet) — keep the SQL-injection callout.


## Phase 6 — Vulnerability Preservation Audit

This phase confirms the **other seven** intentional vulnerabilities still fire. It's read-only — no edits.

### 6.1 VULN-1 SQL Injection Still Possible (AC-07, TC-09)

The username branch is still string-concatenated. Demonstrate with the comment-truncation payload:

```bash
curl -s -i -X POST http://localhost:3001/login \
     --data-urlencode "username=alice' --" \
     --data-urlencode 'password=pass123'
```

Expected behavior:

- The SQL becomes `SELECT * FROM users WHERE username = 'alice' --'`. The `--` comments out the trailing apostrophe.
- `alice`'s row is fetched.
- `verify_password("pass123", alice.password)` returns `True`.
- HTTP 200 with success JSON.

This is the proof that the SQL injection vector is still present (the original query was altered by user input), even though the password verification has moved to Python. A future "SQLi fix" task will parameterize this query.

Note: a stronger demonstration — `username=' OR '1'='1' --` — will fetch the **first** user row but then fail at `verify_password` because the attacker doesn't know that user's password. The fact that the SQL was altered is the vulnerability; bcrypt incidentally narrows the *impact*, but the underlying injection class is unchanged.

### 6.6 VULN-7 No Rate Limiting


## Rollback Procedure

If a phase fails verification and cannot be repaired quickly:

```bash
git restore backend/app/core/security.py
git restore backend/app/services/auth_service.py
git restore backend/pyproject.toml
git restore pyproject.toml
cd backend && uv sync && cd ..
rm -f vulnerable_app.db
```

The four authored files snap back to their pre-fix state, the lock regenerates without bcrypt, and the DB is reset clean. No data migration is involved (the DB is throwaway lab data).

## Out-of-Band: What This Plan Deliberately Does NOT Do

To make the negative space explicit:

- **No SQL parameterization.** The username branch of `login()` and the full `INSERT` in `signup()` stay string-concatenated. VULN-1 remains.
- **No on-login MD5 upgrade.** Legacy rows are unauthenticatable by design (spec §11). The migration path is "reset the DB" or "re-register," never "transparently upgrade."
- **No timing-attack mitigation.** No artificial sleep, no `secrets.compare_digest`, no constant-time username lookup. The spec does not require it; adding it risks scope creep into VULN-7-adjacent territory.
- **No logging of failed attempts.** That would interact with VULN-7 (rate limiting / monitoring) and is a separate concern.
- **No password-strength meter, no zxcvbn, no minimum length.** Out of scope. The educational lab continues to accept any non-empty password.
- **No schema change.** `users.password TEXT` already accommodates a 60-char bcrypt string. No `ALTER TABLE`.
- **No new file.** Spec §3 names four affected files; the plan honors that. No `migrations/`, no `scripts/`, no `tests/`.
